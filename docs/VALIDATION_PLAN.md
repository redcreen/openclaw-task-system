# OpenClaw Task System Validation Plan

## 1. 验证目标

验证工作的目的不是只证明“脚本能运行”，而是证明系统真的改善了长任务体验。

因此验证分为两类：

- 功能验证：系统是否按设计工作
- 体验验证：用户是否不再频繁遇到“催一下才动一下”

## 2. 验证分层

### 2.1 单元测试

验证最小功能单元：

- task schema 与状态读写
- 状态迁移
- 时间戳更新
- 静默超时判断
- outbox payload 生成

### 2.2 脚本级测试

验证脚本行为：

- 扫描脚本能发现超时任务
- outbox 消费脚本能推动事件流转
- 归档逻辑能正确搬移任务

### 2.3 集成测试

验证最小闭环：

- 创建任务
- 推进任务
- 模拟静默
- 静默检测模块发现超时
- 发送兜底事件
- 任务终态收口

### 2.4 体验验收

从使用者视角验证：

- 长任务是否能被持续跟踪
- 任务静默时是否会收到保守同步
- 任务阻塞时是否能看到真实原因
- 用户再次追问时是否基于已有任务状态，而不是重新开始

## 3. 关键验证场景

### 场景 A：正常推进

预期：

- 任务被注册
- 状态进入 `running`
- 每次阶段更新后 `last_user_visible_update_at` 变化
- 最终进入 `done`

### 场景 B：超时静默

预期：

- 任务处于 `running`
- 超过静默阈值
- 静默检测模块识别命中
- 产出保守通知事件

### 场景 C：阻塞任务

预期：

- 任务进入 `blocked`
- 静默检测模块不再按正常运行任务处理
- 状态中保留阻塞原因

### 场景 D：失败任务

预期：

- 任务进入 `failed`
- 不再继续留在正常 inflight 处理中

### 场景 E：并发任务

预期：

- 多个任务互不串线
- 不同 agent / session / chat 的任务不会相互污染

## 4. 自动化测试通过标准

最终自动化测试通过至少应覆盖：

- task state 单元测试
- 静默扫描单元测试
- outbox / send 链路脚本测试
- `main` 最小闭环集成测试

通过标准：

- 所有核心测试可一条命令执行
- 全部通过
- 不需要人工修改测试数据后再跑

## 5. 验收标准

系统达到以下条件，可视为第一阶段通过：

- `main` 的长任务能被注册与追踪
- 超时后有保守同步机制
- 完成 / 阻塞 / 失败都有显式终态
- 自动化测试稳定通过

## 6. 不通过判定

以下任一情况都不能算通过：

- task 状态和用户可见状态明显不一致
- 静默通知模块会重复轰炸同一任务
- 已完成任务仍被继续扫描
- 多任务并发出现串会话 / 串 agent
- 只能手工演示，无法自动回归
