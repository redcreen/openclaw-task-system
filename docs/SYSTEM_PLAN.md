# OpenClaw Configurable Task System Plan

## 1. 系统目标

本系统的目标是构建一个 OpenClaw 可配置任务管理系统。

它服务的不是单个提醒场景，而是 OpenClaw 中所有长任务、多步骤任务、跨轮次任务的统一生命周期管理。

最终目标包括：

- 让任务成为系统中的独立对象，而不是附着在聊天轮次上的临时行为
- 让 agent 的“正在执行 / 已阻塞 / 已完成”有真实状态支撑
- 让长任务不再主要依赖用户催促才能继续推进
- 让 OpenClaw 能通过配置方式启用、禁用、调整任务管理能力
- 让 OpenClaw 能通过插件方式接入，而不是修改主程序
- 让系统具备自动测试与回归能力

## 2. 需要解决的核心问题

当前 OpenClaw 在长任务上存在几类典型问题：

- 任务状态真假混杂：agent 说“在做”，但真实执行未必持续
- 任务静默失联：超过 30 秒没有用户可见进展
- 任务续航依赖用户催促：用户再发一条消息，任务才重新推进
- 任务缺少统一真相源：只能靠聊天记录猜状态
- 多 agent / 多会话任务没有统一管理模型

这些问题本质上不是“提示词写得不够严”，而是系统缺少一层面向任务的运行时。

## 3. 设计边界

本系统应负责：

- 定义任务实例
- 存储任务状态
- 更新任务状态
- 检测任务静默
- 触发兜底通知
- 归档任务终态
- 提供接入 API 和配置项
- 提供插件接入层

本系统当前阶段不直接负责：

- 具体业务逻辑执行
- 所有外部平台消息发送的最终实现
- 智能推断业务进展
- 复杂自动恢复策略的第一版全量实现

## 4. 系统设计思路

### 4.1 总原则

从“消息驱动”补成“任务驱动”。

即：

- 用户消息可以触发任务
- 但任务一旦创建，就应该拥有独立状态与生命周期
- 后续系统行为应尽量围绕任务实例而不是单轮对话

### 4.2 四层结构

#### 第一层：规则层

作用：

- 规定状态词的真实含义
- 限制模糊进度表达
- 要求长任务分阶段同步

可复用现有资产：

- `workspace/shared/EXECUTION_RULES.md`
- `workspace/shared/TASK_EXECUTION_CHECKLIST.md`

#### 第二层：任务状态层

作用：

- 定义 task instance
- 持久化任务状态
- 提供统一读写 API

建议最少状态：

- `queued`
- `running`
- `blocked`
- `paused`
- `done`
- `failed`

#### 第三层：静默检测与通知层

作用：

- 扫描任务静默情况
- 根据阈值决定是否需要兜底同步
- 产出保守通知事件

#### 第四层：接入与配置层

作用：

- 提供给 OpenClaw 的插件入口与配置入口
- 决定哪些 agent 启用任务系统
- 决定静默阈值、通知策略、归档策略、发送路径

## 5. 建议的配置模型

建议未来提供统一配置块，例如：

```json
{
  "taskSystem": {
    "enabled": true,
    "storageDir": "/path/to/task-system/data",
    "agents": {
      "main": {
        "enabled": true,
        "longTaskThresholdSeconds": 30,
        "silenceMonitor": {
          "enabled": true,
          "silentTimeoutSeconds": 30,
          "resendIntervalSeconds": 30
        }
      }
    },
    "delivery": {
      "mode": "session-aware"
    }
  }
}
```

第一版不必一次把所有配置都实现完，但设计上应预留：

- 系统级开关
- agent 级开关
- 超时阈值
- 重发阈值
- 存储路径
- 发送模式

## 6. 当前实现基线

当前 `scripts/runtime/` 中已经具备若干可继续发展的基础实现，包括：

- 任务状态雏形
- 静默扫描雏形
- outbox / 发送准备雏形

这些实现应当被视为本项目当前的运行时基线，而不是外部历史包袱。

## 7. 第一阶段目标

第一阶段不是做全功能系统，而是做“最小可运行闭环”。

第一阶段必须实现：

- `main` 可创建任务实例
- `main` 的阶段同步能回写任务状态
- 静默检测模块可扫描静默任务
- 静默检测模块可产出保守通知事件
- 任务完成 / 阻塞 / 失败有终态收口
- 有自动化测试入口
- 有插件优先的 OpenClaw 接入骨架

## 8. 第二阶段目标

第二阶段再继续推进：

- 多 agent 接入
- 更清晰的配置模型
- 更稳定的真实发送路径
- 恢复策略
- 多任务并发治理

## 9. 成功标准

本系统成功的标志不是“多了几个脚本”，而是：

- 用户不再频繁遇到“催一下才动一下”
- agent 的状态表达和系统内部状态基本一致
- 任务可以被创建、更新、检测、通知、收口
- 核心流程可以自动测试通过

## 10. 当前结论

这不是单纯在做单点提醒能力，而是在为 OpenClaw 建立一个真正的任务运行层。

最终产物应是：

> 一个 OpenClaw 可配置、可插件化接入的任务管理系统
